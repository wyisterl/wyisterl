前言

随着微服务体系架构的日益壮大，传统强大的监控如 Zabbix 对容器监控的问题稍显力不从心，为了解决监控容器的问题， Prometheus 逐渐进入大家的视野中。Prometheus 拥有着易于管理、轻易能够获取服务内部状态、支持本地和远程存储等一系列优点，广受大家欢迎，但需要指出的是，由于数据采集可能会有丢失，所以 Prometheus 不适用对采集数据要 100% 准确的情形。但如果用于记录时间序列数据，Prometheus 具有很大的查询优势。本文旨在介绍 Prometheus 的部署及基本配置。

简介

Prometheus 是一个开源的监控和警报系统，它直接从目标主机上运行的代理程序中抓取指标，并将收集的样本集中存储在其服务器上，是由前 Google 工程师从 2012 年开始在 Soundcloud 以开源软件的形式进行研发的系统监控和告警工具包，自此以后，许多公司和组织都采用了 Prometheus 作为监控告警工具。Prometheus 的开发者和用户社区非常活跃，它现在是一个独立的开源项目，可以独立于任何公司进行维护。为了证明这一点，Prometheus 于 2016 年 5 月加入 CNCF 基金会，成为继 Kubernetes 之后的第二个 CNCF 托管项目。

Prometheus 大多由以下部分组成：

  • Prometheus Server: 用于收集和存储时间序列数据。
  • Client Library: 客户端库，为需要监控的服务生成相应的 metrics 并暴露给 Prometheus server。当 Prometheus server 来 pull 时，直接返回实时状态的 metrics。
  • Push Gateway: 主要用于短期的 jobs。由于这类 jobs 存在时间较短，可能在 Prometheus 来 pull 之前就消失了。为此，这次 jobs 可以直接向 Prometheus server 端推送它们的 metrics。这种方式主要用于服务层面的 metrics，对于机器层面的 metrices，需要使用 node exporter。
  • Exporters: 用于暴露已有的第三方服务的 metrics 给 Prometheus。
  • Alertmanager: 从 Prometheus server 端接收到 alerts 后，会进行去除重复数据，分组，并路由到对收的接受方式，发出报警。常见的接收方式有：电子邮件，pagerduty，OpsGenie, webhook 等。

配置prometheus

附上 Prometheus 官方配置说明文档地址：https://prometheus.io/docs/prometheus/latest/configuration/configuration/

下面会逐一讲解 Prometheus 的核心配置，示例中用到的各种占位符包括：

参数 含义
<boolean> 布尔值，true 或 false
<duration> 持续时间，格式符合正则表达式 [0-9]+(ms
<labelname> 标签名，格式符合正则表达式 [a-zA-Z_][a-zA-Z0-9_]*
<labelvalue> 标签值，可以包含任意 unicode 字符
<filename> 文件名，任意有效的文件路径
<host> 主机，可以是主机名或 IP，后面可跟端口号
<path> URL路径
<schema> 协议，http 或 https
<string> 字符串
<secret> 密钥，比如密码
<tmpl_string> 模板字符串，里面包含需要展开的变量

Prometheus 的配置文件格式为 yaml ，下面是一个配置文件所有参数项说明的示例

全局配置

Global属于全局的默认配置，对所有其它节点都有效，同时也是其它节点的默认值，下列是常见配置项：

  • scrape_interval: 拉取 targets 的默认时间间隔。
  • scrape_timeout: 拉取 target 的超时时间。
  • evaluation_interval: 执行 rules 的时间间隔。
  • external_labels: 额外的标签，会添加到拉取的数据并存到数据库中。
  • query_log_file: 所使用的PromQL会存储到该文件中。

数据拉取配置

Scrape_configs主要用于配置拉取数据节点，每一个拉取配置主要包含以下参数：

  • job_name：任务名称
  • scrape_interval： 拉取时间间隔
  • scrape_timeout: 拉取超时时间
  • metrics_path： 拉取节点的 metric 路径
  • honor_labels： 用于解决拉取数据标签有冲突，当设置为 true, 以拉取数据为准，否则以服务配置为准
  • honor_timestamps： 用于解决拉取数据时间戳有冲突，当设置为 true, 以拉取数据为准，否则以服务配置为准
  • scheme： 拉取数据访问协议
  • params：数据拉取访问时带的请求参数
  • basic_auth： 适用于对Prometheus添加认证的情况，需要通过认证才可拉取
  • bearer_token： 适用于对Prometheus添加认证的情况，需要设置相应的token才可拉取，与bearer_token_file互斥
  • bearer_token_file： 适用于对Prometheus添加认证的情况，需要设置相应的token才可拉取，与bearer_token互斥
  • sample_limit： 存储的数据标签个数限制，如果超过限制，该数据将被忽略，不入存储；默认值为0，表示没有限制
  • relabel_configs： 拉取数据重置标签配置
  • metric_relabel_configs：metric 重置标签配置
